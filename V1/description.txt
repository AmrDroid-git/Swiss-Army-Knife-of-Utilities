# Context
Act as an expert Python software architect and PySide6 developer. I am building a highly flexible, component-based "Visual Scripting GUI Builder." 

# Project Overview
The software allows users to create custom GUI windows visually, drag and drop components, and link those components to external Python scripts. It operates similarly to a simplified QtDesigner or a game engine editor. The application has two main states: "Edit Mode" (for dragging, dropping, and configuring widgets) and "Run Mode" (for locking the UI and executing the linked Python scripts).

# Core Architecture & Data Flow
The application uses a strict Data-Driven Property System. The core engine knows nothing about the specific contents of a window until it reads a `config.json` file. Each window the user creates is saved as a completely self-contained "bundle" (a folder containing its config, custom python scripts, and output assets).

# Key Features & Mechanics
1. Edit Mode vs. Run Mode: 
   - In "Edit Mode", `dynamic_canvas.py` shows the `edit_palette.py`. Widgets allow drag-and-drop. Right-clicking a widget opens a context menu to change its properties (e.g., change arg_order, link a script, delete widget).
   - In "Run Mode", drag-and-drop is disabled. The UI acts as a normal software interface.

2. Widget Properties (JSON Driven) & Context Menus:
   - All widgets inherit from `base_widget.py`. 
   - `base_widget.py` handles the core `contextMenuEvent` for basic actions like Delete. Subclasses append their own specific actions to this menu (like "Set Arg Order" for `widget_i_text.py`).
   - They ingest a `properties` dictionary from `config.json`. This allows extreme flexibility (e.g., if "text_color" is in the JSON, the widget applies it via StyleSheet).

3. Argument Ordering & Script Execution:
   - When a user clicks a `widget_button.py` in Run Mode, the button scans the canvas for all input widgets (the `_i_` widgets).
   - It sorts these inputs based on their `arg_order` property.
   - It sends the sorted values and the linked script path to the `script_engine.py`.
   - `script_engine.py` runs the script using PySide's `QProcess`. It catches `readyReadStandardOutput` and pipes the prints to the `widget_console.py` for real-time terminal output.

# Constraints & Rules for AI
- All UI code must use PySide6.
- The UI must NEVER freeze during script execution. Always use `QProcess` in the script engine, never `os.system` or standard `subprocess.run` without threads.
- Widgets must be modular. They must serialize to and from dictionary formats (`to_dict()` and `from_dict()`) for easy JSON saving/loading.
